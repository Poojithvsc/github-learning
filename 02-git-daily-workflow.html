Git Daily Workflow

This section covers the commands you'll use every single day. Master these, and you'll handle 90% of your Git interactions confidently.

The Basic Workflow Cycle

Edit → Stage → Commit → Push

1. Make changes to files

↓

2. Stage changes you want to commit

↓

3. Commit staged changes (create snapshot)

↓

4. Push to remote (share with team)

git status - Your Best Friend

Always run git status first. It tells you exactly what's happening in your repository.

git status

Sample output:

On branch main
Your branch is up to date with 'origin/main'.

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   src/app.js

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        src/utils.js

no changes added to commit (use "git add")

Understanding File States

State Meaning Color in status
Untracked New file Git doesn't know about Red
Modified Changed file, not yet staged Red
Staged Ready to be committed Green
Committed Safely stored in repository Not shown (clean)
Pro tip: Use git status -s for a shorter output: 
 M src/app.js      # Modified, not staged
M  src/index.js    # Modified and staged
?? src/utils.js    # Untracked
A  src/new.js      # New file, staged

git add - Staging Changes

The git add command moves changes from your working directory to the staging area.

Common Usage

# Add a specific file
git add filename.js

# Add multiple specific files
git add file1.js file2.js file3.js

# Add all files in a directory
git add src/

# Add all changed files (most common)
git add .

# Add all tracked files that were modified (ignores new files)
git add -u

git add -p - Selective/Interactive Staging

This is a powerful feature! It lets you stage only parts of a file.

git add -p

Git will show you each change (called a "hunk") and ask what to do:

@@ -1,4 +1,5 @@
 function greet() {
-    console.log("Hello");
+    console.log("Hello, World!");
+    return true;
 }

Stage this hunk [y,n,q,a,d,s,e,?]?
Option Action
yYes, stage this hunk
nNo, don't stage this hunk
qQuit (don't stage remaining hunks)
aStage this and all remaining hunks in the file
dDon't stage this or any remaining hunks in the file
sSplit this hunk into smaller hunks
eManually edit this hunk
Why use git add -p?
  • Keep commits focused on one logical change
  • Review your changes before committing
  • Avoid accidentally committing debug code
  • Separate unrelated changes into different commits

git commit - Creating Snapshots

A commit saves your staged changes permanently in the repository.

# Commit with a message (most common)
git commit -m "Add user authentication feature"

# Commit with a detailed message (opens editor)
git commit

# Stage all tracked files AND commit (skip git add for modified files)
git commit -am "Fix login bug"
Warning: git commit -am only stages modified tracked files. New (untracked) files won't be included!

Writing Good Commit Messages

A good commit message explains what changed and why.

The 50/72 Rule:
  • First line: 50 characters or less (summary)
  • Blank line
  • Body: Wrap at 72 characters (detailed explanation)
# Bad commit messages
git commit -m "fix"
git commit -m "update"
git commit -m "asdfasdf"
git commit -m "WIP"

# Good commit messages
git commit -m "Fix null pointer exception in user login"
git commit -m "Add email validation to registration form"
git commit -m "Refactor database connection for better performance"

Detailed Commit Message Example

git commit

This opens your editor where you can write:

Add password strength indicator to registration

- Display strength meter below password field
- Color-coded feedback (red/yellow/green)
- Check for minimum length, numbers, and special characters
- Show requirements list that updates as user types

Closes #234

git pull - Getting Updates

git pull downloads changes from the remote repository and integrates them into your local branch.

# Basic pull (fetch + merge)
git pull

# Pull with rebase (cleaner history)
git pull --rebase

# Pull from a specific remote and branch
git pull origin main

git pull vs git pull --rebase

Command What it does Result
git pull Fetch + Merge Creates a merge commit if branches diverged
git pull --rebase Fetch + Rebase Replays your commits on top of remote (linear history)
Recommendation: Use git pull --rebase for a cleaner, linear history. You can set it as default: 
git config --global pull.rebase true

git push - Sharing Your Work

git push uploads your local commits to the remote repository.

# Push to the tracked remote branch
git push

# Push to a specific remote and branch
git push origin main

# Push a new branch and set up tracking
git push -u origin feature/new-login

# Push all branches
git push --all
Never use git push --force on shared branches! It rewrites history and can cause teammates to lose work. If you must force push, use git push --force-with-lease which is safer.

.gitignore - Excluding Files

The .gitignore file tells Git which files to ignore (not track).

Common .gitignore patterns

# Dependencies
node_modules/
vendor/
__pycache__/

# Build outputs
dist/
build/
*.exe
*.dll

# IDE and editor files
.idea/
.vscode/
*.swp
*.swo

# Environment and secrets
.env
.env.local
*.pem
credentials.json

# OS files
.DS_Store
Thumbs.db

# Logs
*.log
logs/

# Temporary files
*.tmp
*.temp
Tip: Use gitignore.io to generate .gitignore files for your specific tech stack.

Ignoring Already Tracked Files

If you've already committed a file and want to ignore it:

# Remove from Git but keep on disk
git rm --cached filename.txt

# Then add to .gitignore and commit

Complete Daily Workflow Example

# Start your day: get latest changes
git pull --rebase

# Check current state
git status

# Make your changes to files...
# (edit code in your editor)

# See what you changed
git status
git diff

# Stage changes (selectively or all)
git add -p
# or: git add .

# Commit with a clear message
git commit -m "Add email validation to signup form"

# Push to remote
git push

# If push is rejected (someone pushed first):
git pull --rebase
git push

Quick Reference

Command Purpose
git status See current state of files
git add <file> Stage specific file
git add . Stage all changes
git add -p Interactive staging
git commit -m "msg" Commit with message
git pull --rebase Get remote changes cleanly
git push Share commits to remote
git diff See unstaged changes
git diff --staged See staged changes

Key Takeaways

  1. Always check status first: git status before and after operations
  2. Stage intentionally: Use git add -p for precise control
  3. Write meaningful commits: Future you will thank present you
  4. Pull before push: Always sync with remote before pushing
  5. Use .gitignore: Keep secrets and junk out of your repository
