Git - The Feynman Way

"If you can't explain it simply, you don't understand it well enough."

This document explains Git using the Feynman Technique: simple words, clear analogies, and zero jargon. Read this when you want to truly understand Git, not just memorize commands.

What is Git, Really?

Explain Like I'm 5:

Imagine you're writing a story. Every time you finish a chapter, you make a photocopy and put it in a folder. Now you can always go back to any chapter if you mess up. Git does this automatically for your code.

Git is a tool that takes snapshots of your project every time you tell it to. Each snapshot is called a commit. You can go back to any snapshot at any time.

The Video Game Analogy

Git is like save points in a video game. Every time you complete something important, you save. If you die (mess up your code), you load your last save. Git lets you have unlimited saves and jump to any of them.

The Three Boxes

Git has three "boxes" where your files can be:

Working Directory → Staging Area → Repository

(Your desk) → (Outbox) → (Filing cabinet)
The Office Analogy

Working Directory: Your desk where you're actively working on papers.

Staging Area: Your outbox - papers you've finished and want to file.

Repository: The filing cabinet - permanently stored, organized records.

In Simple Terms:
1. You edit files (working directory)
2. You choose which edits to save: git add (staging)
3. You save them permanently: git commit (repository)

Commits = Snapshots

Explain Like I'm 5:

A commit is like taking a photo of your entire Lego creation. The photo remembers exactly how every piece was arranged at that moment. You can take hundreds of photos as you build, and look back at any of them.

Each commit contains:

  • A snapshot of ALL your files at that moment
  • A unique ID (like a fingerprint)
  • Who made it and when
  • A message explaining what changed
  • A pointer to the previous commit (its parent)
Key Insight: Commits are SNAPSHOTS, not differences. Git doesn't store "what changed" - it stores "what everything looks like right now."

Branches = Parallel Universes

The Multiverse Analogy

Imagine you're at a crossroads in life. In one universe, you take the left path. In another, you take the right. Both versions of you continue living, making different choices.

Git branches are like parallel universes for your code. You can work on a new feature in one universe while keeping your main code safe in another. Later, you can merge the universes together.

What a branch actually is:
A branch is just a sticky note pointing to a commit. That's it! When you make new commits, the sticky note moves forward. Creating a branch just creates a new sticky note.
git switch -c new-feature # Create new universe
git switch main # Jump back to main universe

HEAD = "You Are Here"

Explain Like I'm 5:

You know those maps at the mall with "You Are Here" arrows? HEAD is Git's way of showing "You Are Here" - which branch you're on and which commit you're looking at.

The Daily Workflow - Super Simple

Every Day, Every Project:

git status # Where am I? What changed?
git add . # Stage all changes
git commit -m "What I did" # Save snapshot
git push # Share with team
The Restaurant Analogy

git status = Looking at your order pad

git add = Writing down what the customer wants

git commit = Sending the order to the kitchen

git push = Serving the food to the customer

Merging = Combining Universes

Explain Like I'm 5:

You and your friend both drew parts of a picture. Now you want to combine them into one picture. If you drew different parts, it's easy - just put them together. But if you both colored the same spot differently, you have to decide whose color to keep. That decision is a "merge conflict."

Merging: Combining two branches into one.
Conflict: When both branches changed the same line - Git needs your help to decide which version to keep.

Remote = The Cloud Copy

The Google Docs Analogy

Your local repository is like a Word document on your computer. The remote (GitHub) is like Google Docs - it's in the cloud, and your whole team can access it.

git push = Uploading your document to Google Docs

git pull = Downloading the latest version from Google Docs

git fetch = Checking if there are new versions (without downloading yet)

Undo Anything - Git is Forgiving

The Undo Cheat Sheet (memorize this!)

Oops, I want to discard my uncommitted changes:
→ git restore filename

Oops, I staged something I didn't want to:
→ git restore --staged filename

Oops, I committed but want to undo (locally):
→ git reset --soft HEAD~1 (keeps changes staged)
→ git reset HEAD~1 (keeps changes unstaged)
→ git reset --hard HEAD~1 (discards everything)

Oops, I pushed something bad:
→ git revert <commit> (creates an "undo" commit - safe)

Oops, I really messed up and can't find my work:
→ git reflog (shows everything that happened - your time machine)

Golden Rule: If it's committed, it's almost never lost. Use git reflog to find it.

The 5 Commands That Cover 90% of Git

  1. git status - What's going on?
  2. git add - I want to save these changes
  3. git commit - Take the snapshot
  4. git pull - Get team's changes
  5. git push - Share my changes

One Last Analogy: The Complete Picture

Git is a Photo Album for Your Code

Repository: Your photo album

Commits: Individual photographs

Branches: Different sections of the album (vacations, birthdays, etc.)

HEAD: The page you're currently looking at

Staging: Choosing which photos to put on the next page

Remote: A backup copy of your album in the cloud

Push/Pull: Syncing your album with the cloud copy

Merge: Combining two sections into one

Conflict: Two photos in the same spot - you pick which one

Reflog: An index showing every page you've ever looked at

Remember: Git's job is simple - save snapshots of your work and let you travel back to any of them. Everything else is just a convenience built on top of that.
